# Prizma — спецификация панели управления и поставки

Документ описывает, **как продукт передаётся клиенту** и **как выглядит и работает
панель управления**. Дополняет [TZ.md](TZ.md) (цель и требования к сбору) и
[HANDOVER.md](HANDOVER.md) (доказанные методы). Написано для владельца; технические
детали — в объёме, достаточном для постановки задачи разработчику.

> **Важно (расхождения с текущим ТЗ):** в `TZ.md` в разделе «Не делаем» указано
> «не строим UI/дашборд», а метод парсинга Ozon в `ADR-0002` завязан на headless-Playwright.
> Оба пункта **пересматриваются** этим документом: панель управления теперь **в объёме
> проекта**, а рабочий метод Ozon — прогретые куки + `curl_cffi` (см. HANDOVER, п.3), а
> Playwright остаётся только для разового прогрева кук. Нужно обновить `TZ.md` и завести
> новый ADR по методу сбора.

## 1. Модель поставки

Продукт передаётся **zip-архивом с автоустановщиком**. Пользователь распаковывает и
запускает установщик, который сам:

- проверяет/ставит Python и системные зависимости;
- создаёт изолированное окружение и ставит все библиотеки (Playwright+Chromium,
  `curl_cffi`, драйвер PostgreSQL и т.д.);
- разворачивает скрипты проекта и **локальную веб-панель**;
- создаёт ярлык/команду запуска.

**Целевые среды — обе:**

- **Windows (локально)** — установщик `.exe`/`.bat`, панель поднимается локально.
- **Linux (сервер)** — установочный скрипт (или Docker/systemd-сервис).

**Что такое «панель».** Это локальная веб-панель: установщик поднимает на машине
маленький веб-сервер, пользователь открывает в браузере адрес вида
`http://127.0.0.1:8000` и работает в нём. Одинаково работает и на локальной машине, и на
сервере (на сервере — с входом по паролю). Отдельную программу ставить не нужно —
достаточно браузера.

> Особенность: браузерный **прогрев кук** (см. вкладку «Куки») требует видимого браузера,
> что естественно на локальной Windows-машине; на «голом» Linux-сервере без графики этот
> шаг требует отдельного решения (прогрев на локальной машине и перенос кук, либо
> удалённый браузер). Помечено в открытых вопросах.

## 2. Общая структура панели

Сверху — **горизонтальные вкладки**. Визуальный стиль — по **бренд-буку Vector** (цвета,
шрифты, логотип, компоненты; файл бренд-бука — предоставляет владелец, см. открытые
вопросы). Вкладки:

1. Панель управления
2. Куки
3. Параметры подключения
4. Редактор скриптов
5. Логи / история

## 3. Вкладка «Панель управления» (дашборд)

Общее состояние проекта одним взглядом.

> **Блок «Города» — реализован** (`prompt-14-cities-tab`, [ADR-0011](adr/0011-local-settings-store.md)):
> добавление города, «Наследует общее / Локально» + прокси/интервал/вкл-выкл по каждой
> площадке отдельно, удаление (деактивация) города — всё через локальный стор
> `config/cities.json`, движок читает работу через него. Блоки «WB»/«OZ» (общий профиль) и
> глобальный тумблер площадки ниже — пока **read-only показ**, редактор — следующий слайс.

- **Здоровье проекта:** работает / не работает; когда был последний запуск; итог
  последнего прогона (успешно / с ошибками / заблокировано), доля успеха.
- **Кнопка «Запустить сейчас»** — весь прогон, либо выборочно по площадке/городу.
**Модель настроек — «общее + локальное переопределение».**

- **Блоки «WB» и «OZ» — общий профиль подключения (по умолчанию).** Здесь задаётся общее
  подключение на каждую площадку: прокси, куки (если нужны), интервал обновления, прочие
  параметры. Кнопка **проверки прокси** (IP / регион / оператор). Эти значения применяются
  ко всем городам, пока город не переопределит их локально. Модели разные: WB — прокси на
  каждый замер; Ozon — прокси в основном для прогрева кук.
  - **Тумблер вкл/откл площадки (глобально)** — выключенную площадку не мониторим нигде
    (напр., владельцу нужны цены только одного маркетплейса, а не сравнение между ними).
- **Блок «Города (+)» — точечная настройка.** Кнопка «+» добавляет город. Каждый город
  **по умолчанию наследует общий профиль**, но при необходимости **локально переопределяет**
  прокси / куки / интервал / прочее — отдельно для WB и/или OZ (лечим точечно по паре
  «город × площадка»).
  - **Свёрнутый город:** название, индикатор здоровья, время последнего запуска и минимум
    полезного (статус WB/OZ, доля успеха).
  - **Раскрытый город:** под-блоки WB и OZ с переключателем «Наследует общее / Локально»;
    в режиме «Локально» — свои поля прокси/куки и т.д.; кнопки «проверить прокси»,
    «обновить куки», «тест региона». Место под будущие параметры (расширяемо).
  - **Тумблер вкл/откл площадки в городе** (переопределяет общий) — можно отключить,
    например, только Ozon и только в этом городе, оставив WB.
  - **Пометка на городе:** «наследует общее» либо «есть локальные изменения».
- **Общие параметры** — значения по умолчанию для всего проекта: базовое расписание,
  темп/лимиты запросов, таймауты, подключение к БД, каналы уведомлений, срок хранения
  истории и логов.
- **Уведомления/алерты** — в Telegram или на почту: падение прогона, протухание кук,
  мёртвый прокси, просадка доли успеха ниже порога.

## 4. Вкладка «Куки» — ✅ доставлено локально (`prompt-16-ozon-address-memory`, ADR-0012/0013)

Цель — **максимально автоматизировать** получение кук, оставив пользователю простой
ручной контроль.

- **«Авторизоваться и собрать»** (главная кнопка, симметрична для WB и Ozon): оператор
  логинится/проходит защиту **один раз** в открывшемся видимом браузере, дальше поток сам
  обходит настроенные города («Города», §3), останавливаясь только на капче (авто-детект +
  опрос, без консоли). Под капотом асимметрично:
  - **Ozon** — регион зашит **в куке**, которая пишется только когда выбран адрес доставки в
    UI (не по имени города — это была неверная модель, ADR-0013). Кнопка проходит **guided**
    обход городов **без запомненного адреса**: оператор выбирает/подтверждает адрес в
    видимом окне, поток запоминает `address_label` вместе с кукой на этот город.
  - **WB** — регион едет через прокси ⇒ **одна** сессионная кука (`_session`), не по городам.
- **«Обновить протухшие»** (Ozon): авто-починка — для городов, у которых кука протухла и уже
  есть запомненный адрес, поток авто-переизбирает этот адрес из адресной книги аккаунта и
  сохраняет свежую куку без участия оператора (браузер выводится на экран только на капчу
  или если нужен повторный логин).
- **Прогресс вживую** в панели: текущий город, «сохранено» / «ждём капчу» / «готово» — через
  поллинг статуса фоновой джобы (общий для collect и refresh).
- **Ручное управление:** пользователь видит текущие куки (и запомненный адрес Ozon) и может
  их отредактировать/вставить вручную (включая адрес), либо сбросить.
- **По регионам:** куки Ozon привязаны к региону — храним набор кук + запомненный адрес на
  каждый активный город; WB — один сессионный набор.
- **Безопасность:** локальный однопользовательский режим (ADR-0009) — куки хранятся в
  открытом виде под `cookie_store_dir`, не в репозитории; шифрование не требуется в этом
  режиме.
- **Здоровье:** возраст кук и статус (валидна/истекает/протухла) через `is_stale` +
  `ozon_cookie_ttl_hours`, плюс запомненный адрес рядом с вердиктом.
- **Открытый вопрос (не реализовано):** прогрев/починка на сервере без экрана
  (headless/Xvfb) — обход сейчас требует локальный видимый браузер (§9.2).

## 5. Вкладка «Параметры подключения»

Слева — **вертикальные закладки**:

### 5.1. Источник данных (список SKU)

Откуда берём список товаров и регионов:

- **Локально:** файл `xlsx`/`csv` — путь к файлу, лист, диапазон.
- **Сервер:** база (PostgreSQL) или REST — параметры подключения (host, port, база,
  логин, пароль / токен, имя таблицы или endpoint).

### 5.2. Сопоставление входа (маппинг источника)

Таблица маппинга: **слева — наши (канонические) поля**, **справа — где их искать в
источнике**.

- Наши поля: `sku`, `url`, `name`, `city/region`, и др. реквизиты (см. п.7).
- Источник: если локально — лист + колонка; если сервер — таблица + колонка (или поле
  ответа).
- **Предпросмотр** на нескольких строках + валидация (чтобы колонки не «уехали»).

### 5.3. Приёмник результатов

Куда складываем замеры — аналогично источнику:

- **Локально:** файл `xlsx`/`csv` (путь, лист).
- **Сервер:** PostgreSQL / REST (параметры подключения, таблица/endpoint).

### 5.4. Сопоставление выхода (маппинг результата)

Таблица маппинга: **слева — наши поля результата** (цена, цена по карте, город, время
замера, статус и т.д.), **справа — куда их писать** (лист+колонка либо таблица+колонка).

## 6. Вкладка «Редактор скриптов»

- Показывает **все скрипты проекта** (Python) списком/деревом.
- Просмотр и **редактирование на лету** прямо в панели, с сохранением.
- Подсветка синтаксиса, кнопка «запустить» скрипт и показать вывод.
- Желательно: резервная копия перед сохранением (откат), пометка изменённых файлов.

> Осторожно: правка боевых скриптов на лету — мощный, но рискованный инструмент. Заложить
> бэкап версии и, при серверной установке, ограничение доступа (только владелец).

> Планируется пайплайн-конструктор поверх независимых скриптов (control-panel, parameters,
> health, wb/ozon, orchestrator) в духе GitHub Actions — см.
> [ADR-0008](adr/0008-script-shell-separation.md) (принято, не реализовано).

## 7. Канонические поля данных

Единый внутренний словарь, к которому маппится вход и от которого маппится выход:

- `marketplace` — `wb` | `ozon`
- `sku` — артикул/ID товара на площадке
- `url` — ссылка на карточку
- `name` — название
- `region` — код/название города
- `price` — текущая цена
- `price_no_card` — цена без карты (перечёркнутая/обычная)
- `price_card` — цена с картой (Ozon-карта / WB-кошелёк)
- `currency` — валюта (`RUB`)
- `availability` — наличие
- `measured_at` — момент замера
- `status` — `ok` | `blocked` | `not_found`

## 8. Связь с движком сбора (что под капотом)

- **WB** — `requests` + региональный прокси.
- **Ozon** — `curl_cffi` (маскировка под Chrome) + прогретые куки; регион задаётся
  региональной кукой; парк региональных прокси для чтения Ozon не обязателен.
- **Прогрев кук** — Playwright + Chromium (вкладка «Куки»), запускается редко.
- Подробности и доказательства — в [HANDOVER.md](HANDOVER.md) и памяти проекта.
