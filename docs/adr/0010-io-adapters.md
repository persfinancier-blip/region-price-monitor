# ADR-0010 — Настраиваемые source/sink-адаптеры (CSV/Excel/DB) + маппинг полей

- **Статус:** принято
- **Дата:** 2026-07-23
- **Контекст задачи:** [ADR-0009](0009-local-first-storage.md) (этот слайс назван там
  следующим), [SPEC-panel.md](../SPEC-panel.md) §5 и §7, [ARCHITECTURE.md](../ARCHITECTURE.md)

## Контекст

ADR-0009 закрыл обязательность Postgres для хранения внутреннего состояния движка, но
источник списка товаров/регионов и приёмник результатов остались жёстко зашитыми:
единственный вход — локальный JSON (`import-products`/`import-regions <file>`), единственный
выход — сам стор. SPEC-panel §5 требует **настраиваемый** источник и приёмник — локальный
файл (`xlsx`/`csv`) или БД/сервер, с **маппингом полей** (наши канонические поля vs
колонки/поля источника) и предпросмотром. Это бэкенд, который вкладка «Параметры подключения»
(Фаза 8.4) позже обернёт UI — здесь строится только движок, конфиг и файлы.

## Решение

1. **I/O seam** (`app/io/`), по стилю зеркальный `app/storage/`:
   - `ProductSource` (`read_products()`/`read_regions()`) и `ResultSink`
     (`write_snapshots(rows)`) — Protocol'ы, оперирующие только **канонически**-ключёванными
     строками (SPEC §7: `marketplace, sku, url, name, region, price, price_no_card,
     price_card, currency, availability, measured_at, status`).
   - `make_product_source(settings)` / `make_result_sink(settings)` выбирают реализацию по
     `kind`: `json` | `csv` | `xlsx` | `db`.

2. **Четыре адаптера:**
   - **`json`** — сегодняшний локальный-JSON путь без изменений (обратная совместимость);
     маппинг не нужен — ключи уже канонические.
   - **`csv`** — `csv`-модуль, маппинг `canonical -> заголовок колонки`.
   - **`xlsx`** — **openpyxl** (новая зависимость), конфиг `path`/`sheet`/опциональный A1
     `range`; маппинг `canonical -> колонка` (имя заголовка или буква).
   - **`db`** — SQLAlchemy (уже в стеке) поверх произвольной таблицы источника/результата;
     креды подключения — в обычном локальном конфиге (никакого хранилища секретов, по
     ADR-0009 п.5); импорт `app.io.db_io` ленивый — файловым бэкендам SQLAlchemy-движок не
     нужен, параллель с `app/storage/postgres.py`.

3. **Маппинг — данные, не код** (`app/io/mapping.py`): `{canonical_field: locator}`,
   загружается из `config/io.json` (путь — `settings.io_config_path`), отдельно секции
   `source` (вход) и `sink` (выход). `validate()` ловит отсутствующие обязательные поля и
   локаторы, которых нет в реальном заголовке источника (съехавший лист, SPEC §5.2).
   `preview(rows, mapping, n=5)` — первые N замаппленных строк для предпросмотра (панель
   отрисует его позже). **Точный список ценовых полей не финализирован (SPEC §9.5)** —
   таблица маппинга ничего не знает про конкретный набор полей результата, поэтому список
   растёт без изменений кода адаптеров.

4. **Тонкие обёртки скриптов (ADR-0008):**
   - `control_panel.py import-products`/`import-regions` — `file` теперь опционален: если
     передан — прежнее поведение один в один; если нет — читает через
     `make_product_source(settings)`.
   - Новый `app/scripts/export.py` (`python -m app.scripts.export`, verb `export` в `cli.py`)
     — строит канонические строки результата из `price_snapshots` (+ join `products`/`regions`
     через storage seam) и пишет через `make_result_sink(settings)`; `--preview` печатает
     первые строки вместо записи; без настроенного стока — чистый no-op с сообщением.

5. **Локально-по-умолчанию:** нет `config/io.json` ⇒ `kind: json`-источник (сегодняшнее
   поведение) и **сток не настроен** (`export` ничего не пишет, сообщает об этом). Пример
   конфига — `config/io.example.json` (csv/xlsx/db формы).

## Что осознанно не входит в этот слайс

- **Мастер настройки (setup wizard)** — `prompt-14`.
- **UI вкладки «Параметры подключения»** (Фаза 8.4) и любые другие вкладки панели.
- **Финализация списка ценовых полей** (SPEC §9.5) — таблица маппинга спроектирована так,
  чтобы список рос без изменений кода, но сам список не определяется здесь.
- Логика коллекторов/прокси/кук/движка замера, поведение storage seam — не тронуты; I/O seam
  добавляет только рёбра чтения-на-входе / записи-на-выходе.

## Обоснование

- Тот же seam-паттерн, что и storage (ADR-0009): Protocol с одинаковой сигнатурой на всех
  бэкендах, движок и скрипты не знают, откуда физически приходят данные.
- Один канонический словарь как единственная внутренняя развязка держит адаптеры
  независимыми друг от друга и от конкретного набора полей результата.
- Маппинг как данные (JSON-конфиг), а не код — то, что панель (Фаза 8.4) будет редактировать
  через UI без переразвёртывания.
- `db`-адаптер ленивый по импорту (как `app/storage/postgres.py`) — файловый бэкенд не тянет
  сетевые зависимости.

## Последствия

- `pyproject.toml` получает `openpyxl` в основные зависимости.
- `.env.example`/`app/config.py` — новая настройка `io_config_path` (по умолчанию
  `config/io.json`), опциональная, ничего не меняет при отсутствии файла.
- Существующий JSON `import-products`/`import-regions` путь и его stdout не изменились;
  все прежние тесты остаются зелёными.
- Следующий слайс: `prompt-14` (мастер настройки поверх этого движка).
