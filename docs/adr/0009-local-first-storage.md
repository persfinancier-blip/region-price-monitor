# ADR-0009 — Local-first storage: Postgres становится опциональным (ревизия ADR-0004)

- **Статус:** принято
- **Дата:** 2026-07-23
- **Контекст задачи:** [TZ.md](../TZ.md), [ADR-0004](0004-scheduling-runtime.md) (частично
  вытесняется этим ADR), [ARCHITECTURE.md](../ARCHITECTURE.md), [SPEC-panel.md](../SPEC-panel.md) §5

## Контекст

Решение владельца (2026-07-23): **отсутствие Postgres не должно блокировать продукт** —
движок обязан работать полностью локально, без базы данных. До этого слайса внутреннее
состояние движка (`products`/`regions`/`runs`/`measure_queue`/`attempts`/`price_snapshots`)
и очередь задач (ADR-0004) жили только в Postgres, что делало БД обязательной для любого
запуска (даже разового `measure-wb` на одном устройстве).

## Решение

1. **Storage seam.** Протоколы репозиториев (`app/storage/base.py`), совпадающие по
   сигнатурам методов с существующими `app/repositories.py`, плюс фабрика
   `make_storage(settings)`, возвращающая связанный набор репозиториев для выбранного бэкенда.

2. **Два бэкенда:**
   - **`local`** (по умолчанию) — плоские файлы (JSON/JSONL) под `settings.local_state_dir`,
     атомарная запись (temp-файл + `rename`), монотонные локальные счётчики id.
   - **`postgres`** — существующие SQLAlchemy-репозитории за тем же интерфейсом (опционально,
     только при `storage_backend=postgres`).

3. **Локальная очередь задач.** `LocalTaskQueue` поверх локального стора: тот же `Protocol`
   (`app/queue/base.py`), что и `PgTaskQueue`. `make_task_queue(settings, storage)` выбирает
   реализацию. Важное отличие: **на локальном бэкенде нет `SELECT ... FOR UPDATE SKIP LOCKED`** —
   это гарантия конкурентности только для Postgres. Для одной машины/одного процесса это не
   проблема; `LocalTaskQueue` не защищён от гонки между независимыми процессами.

4. **Движок ходит только через seam.** `run_once`/`orchestrator`, скрипты `wb`/`ozon`/
   `control_panel`/`health`/`report`, читающий путь панели — все строят репозитории через
   `make_storage(settings)` (и очередь через `make_task_queue`), а не через прямой
   SQLAlchemy-сеанс. `ProxyHealthService` и `compute_run_metrics` читают через тот же seam
   (`attempts.recent_for_proxy_ref`/`attempts.for_run`), а не прямым SQL.

5. **Без хранилища секретов.** Локальный однопользовательский режим: креды прокси/БД лежат в
   обычном локальном конфиге под контролем пользователя — требование шифрования секретов
   (SPEC-panel §9.3) снимается для локального режима; `.env` остаётся как есть.

## Что осознанно не входит в этот слайс

- **Настраиваемые source/sink-адаптеры (CSV/Excel/DB) и маппинг полей** — источник данных
  сейчас — существующий локальный JSON-импорт (`import-products`/`import-regions`), сток —
  локальный стор. Настраиваемые адаптеры + маппинг — `prompt-13`.
- **Мастер настройки (setup wizard)** — `prompt-14`.
- Postgres-путь не удаляется — остаётся полностью рабочим за тем же seam.

## Обоснование

- Seam с одинаковыми Protocol-сигнатурами держит поведение идентичным на обоих бэкендах:
  движок, скрипты и тесты не знают, где физически лежат данные.
- Плоские файлы + атомарная запись достаточны для однопользовательского локального сценария;
  не требуют миграций/сервера БД для старта продукта.
- Явное документирование `SKIP LOCKED` как Postgres-only гарантии избегает ложного ощущения
  безопасности при попытке распараллелить локальный бэкенд между процессами в будущем.

## Последствия

- **ADR-0004 частично вытеснено**: очередь-в-Postgres перестаёт быть единственным вариантом —
  она теперь один из двух `TaskQueue`, выбираемых по `storage_backend`. Роли
  Scheduler/Queue/Worker pool из ADR-0004 сохраняются, только очередь получила локальную
  реализацию.
- Docker-entrypoint запускает `alembic upgrade head` только при `storage_backend=postgres`;
  `docker-compose.prod.yml` делает `postgres`-сервис опциональным (профиль `postgres`), `app`
  работает автономно в локальном режиме.
- Следующие слайсы: `prompt-13` (source/sink адаптеры + маппинг полей, локально или в БД),
  `prompt-14` (мастер настройки/setup helper).
