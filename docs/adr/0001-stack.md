# ADR-0001 — Технологический стек

- **Статус:** принято
- **Дата:** 2026-07-22
- **Контекст задачи:** [TZ.md](../TZ.md)

## Контекст

Нужен сервис на Python (жёсткое требование владельца), который регулярно снимает цены с WB и Ozon в разрезе регионов РФ через прокси, устойчиво проходит антибот и хранит историю в PostgreSQL. Масштаб и хостинг пока не зафиксированы — стек должен стартовать «малой кровью», но масштабироваться без переписывания.

## Решение

- **Python 3.12** — язык проекта.
- **Playwright (Chromium)** как основной механизм сбора — headless-браузер с эмуляцией реального клиента (см. [ADR-0002](0002-scraping-strategy.md)).
- **PostgreSQL 16** — хранилище товаров, регионов и истории замеров.
- **SQLAlchemy 2.x** (async) + **Alembic** — доступ к БД и версионирование схемы.
- **pydantic-settings** — конфигурация из окружения/`.env`.
- **APScheduler** — внутренний планировщик прогонов на MVP (см. [ADR-0004](0004-scheduling-runtime.md)).
- **Docker** + **docker-compose** — упаковка и локальный запуск (app + postgres).
- **Наблюдаемость:** структурированные логи (structlog или loguru); метрики Prometheus — на фазе устойчивости.
- **Качество кода (DoD-гейт):** `ruff` (линт+формат), `mypy` (типы), `pytest` (тесты). Все три собираются в `scripts/dod.sh`.

## Обоснование

- Экосистема Python закрывает и браузерную автоматизацию (Playwright), и работу с Postgres (SQLAlchemy), и планирование — без смены языка.
- Async-стек (asyncio + Playwright async API + SQLAlchemy async) даёт естественный параллелизм замеров с ограничением конкурентности, что прямо нужно для пула прокси/браузерных контекстов.
- Alembic с первого дня избавляет от боли ручных миграций при росте схемы.
- Docker с самого начала снимает вопрос «где запускать» — решение о хостинге откладывается без риска (см. открытые вопросы в ТЗ).

## Альтернативы

- **Scrapy / чистый httpx без браузера** — быстрее и дешевле, но проиграли на антиботе Ozon и региональной достоверности (см. [ADR-0002](0002-scraping-strategy.md)).
- **MongoDB вместо Postgres** — отклонено: владелец задал Postgres, и данные строго реляционные (товары × регионы × замеры во времени).
- **Celery/RQ на старте** — избыточно для MVP; закладываем интерфейс очереди, но включаем распределённые воркеры позже (см. [ADR-0004](0004-scheduling-runtime.md)).

## Последствия

- Браузерный сбор требует больше CPU/RAM и аккуратной конкурентности — учитываем в оркестрации.
- Async-код по всему проекту — единый стиль, но выше порог входа; частично компенсируется типами (mypy) и тестами.
