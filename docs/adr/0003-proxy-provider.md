# ADR-0003 — Прокси: интерфейс ProxyProvider + коммерческий провайдер

- **Статус:** принято
- **Дата:** 2026-07-22
- **Контекст задачи:** [TZ.md](../TZ.md)

## Контекст

Региональная цена достижима только с IP нужного региона. Нужны резидентные/мобильные прокси с гео-таргетингом по РФ. Конкретный вендор пока не выбран, но менять его нельзя ценой переписывания сбора. Прокси — расходуемый и «баняемый» ресурс: нужны ротация, отбраковка и учёт здоровья.

## Решение

Ввести абстракцию **`ProxyProvider`** и кодить весь сбор против неё, а не против конкретного вендора.

Контракт (примерно):

```python
class ProxyProvider(Protocol):
    async def acquire(self, region: RegionCode) -> ProxyLease: ...
    async def report(self, lease: ProxyLease, outcome: Outcome) -> None: ...
```

- `acquire(region)` — выдать рабочий прокси для региона (endpoint, креды, гео-метаданные), с учётом ротации и «остывания» недавно забаненных.
- `report(lease, outcome)` — вернуть исход (ok / soft-ban / hard-ban / timeout) для стратегии здоровья и ротации.
- `ProxyLease` — конкретный прокси на время одной попытки; освобождается/переоценивается после `report`.

Реализации:

- **`CommercialProxyProvider`** — обёртка над API/шлюзом выбранного коммерческого провайдера (резидентные/мобильные, ротация на их стороне или на нашей).
- **`StaticProxyProvider`** (для локали/тестов) — список из конфига, без внешних вызовов.

Гео: справочник `regions` хранит, какой прокси-регион/город запрашивать у провайдера; маппинг «наш регион → параметры провайдера» — часть реализации.

## Обоснование

- Вендор прокси — открытый вопрос ТЗ; интерфейс развязывает выбор вендора и сроки разработки.
- Единая точка для политики здоровья/ротации/бан-листа вместо размазывания по коллекторам.
- Тесты и локальная отладка не зависят от платного провайдера (`StaticProxyProvider`).

## Альтернативы

- **Свой пул прокси-серверов по регионам** — дешевле в эксплуатации, дороже в поддержке; отклонено на старте (владелец выбрал коммерческого провайдера). Легко добавляется как ещё одна реализация `ProxyProvider`.
- **Хардкод вендора прямо в коллекторе** — отклонено: связывает руки при смене вендора и ломает тестируемость.

## Последствия

- Нужны метрики по прокси: доля банов на регион/провайдера, время жизни, стоимость на успешный замер.
- Креды провайдера — только через окружение/секреты, никогда в репозиторий.
- Стратегия ретраев тесно связана с `report()`: hard-ban → сменить прокси, а не долбить тот же.
