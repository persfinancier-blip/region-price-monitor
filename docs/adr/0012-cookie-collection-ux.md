# ADR-0012 — Куки: одна кнопка «Авторизоваться и собрать», авто-обход городов

- **Статус:** принято
- **Дата:** 2026-07-24
- **Контекст задачи:** [SPEC-panel.md](../SPEC-panel.md) §4, [ADR-0005](0005-scraping-method-update.md),
  [ADR-0008](0008-script-shell-separation.md), [ADR-0009](0009-local-first-storage.md),
  [ADR-0011](0011-local-settings-store.md), [ARCHITECTURE.md](../ARCHITECTURE.md),
  [ROADMAP.md](../ROADMAP.md)

## Контекст

Вкладка «Куки» была плейсхолдером (Фаза 8.3 в бэклоге). Решение владельца (2026-07-23):
**одна кнопка на площадку — «Авторизоваться и собрать»** — оператор логинится/проходит
защиту один раз, дальше поток сам обходит настроенные города (`app/scripts/cities.py`,
ADR-0011), сохраняя куку на город, останавливаясь только на капче. Кнопка симметрична для
WB и Ozon, но реализация асимметрична: у **Ozon** регион зашит в куку ⇒ нужна кука **на
каждый город**; у **WB** регион приходит через прокси/запрос ⇒ достаточно **одной сессии**.

## Решение

1. **Симметричный прогрев для обеих площадок.** `app/cookies/warm.py::_MARKETPLACE_WARM_URL`
   расширен записью для WB (`https://www.wildberries.ru/`) — WB становится таким же
   полноценным таргетом прогрева, как Ozon. Существующие `CookieWarmer.warm`/`warm_if_stale`
   не меняются (аддитивно).

2. **`warm_all(store, marketplace, cities)` — логин-раз + авто-обход.** Один видимый
   Playwright-контекст на весь проход:
   - оператор один раз проходит логин/капчу на первой загрузке;
   - **Ozon** — для каждого города из эффективного стора (`cities.list_effective`)
     авто-переключает город доставки (запрос с `?city=` — переписывает региональную куку
     без ручного клика по селектору) и сохраняет **отдельный `CookieBundle` на
     (ozon, город)**; один и тот же залогиненный контекст переиспользуется на все города;
   - **WB** — сохраняет **одну** сессионную куку под служебным кодом `_session` (регион едет
     через прокси, не через куку) — тот же проход, без обхода городов;
   - **Капча — только авто-детект + опрос** (`_looks_like_captcha` ищет маркеры на странице,
     `_wait_out_captcha` поллит до очистки или таймаута) — **никакого консольного `input()`**:
     поток должен управляться из панели, без терминала;
   - у каждого шага свой таймаут (`step_timeout_s`) + кооперативный `cancel`-токен, чтобы
     зависший город не подвесил весь обход; каждый город идёт через свой эффективный прокси
     (warm-IP == fetch-IP, ADR-0005).

3. **Тонкий скрипт** `app/scripts/cookies.py` (ADR-0008): `collect(marketplace)` берёт
   эффективные города из `cities.list_effective` и запускает `warm_all`; `status()` отдаёт
   health (valid/expiring/stale через `is_stale` + `ozon_cookie_ttl_hours`, «expiring» — когда
   до протухания осталось ≤25% TTL); `set_manual`/`clear` — ручная вставка/сброс. Работает и
   отдельно (`python -m app.scripts.cookies`), и под `cli.py cookies ...`.

4. **Панель — тонкая оболочка.** `GET /tab/cookies` рендерит вкладку; `POST
   /cookies/{mp}/collect` стартует фоновую джобу (в отдельном потоке — `asyncio.to_thread`,
   чтобы синхронный Playwright-вызов не блокировал event loop); `GET /cookies/status` отдаёт
   прогресс (город × статус) для поллинга; `POST /cookies/{mp}/{city}` — ручная вставка,
   `POST /cookies/{mp}/{city}/clear` — сброс (помечает bundle protuhшим — отдельного
   delete-метода у `CookieStore` нет). Прогресс джобы живёт в процессе панели
   (`_CollectJob`/`_cookie_jobs`), не персистентен между перезапусками панели — это ок для
   одного локального прогона.

5. **Локально, видимый браузер — сейчас; сервер — открытый вопрос.** Обход требует реальный
   видимый Chromium на локальной машине (как и `CookieWarmer.warm` до этого). Headless/Xvfb
   прогрев на голом Linux-сервере — не реализуется в этом слайсе (SPEC §9.2 остаётся открытым
   вопросом).

6. **Куки — локально, без шифрования** (ADR-0009 §5): `cookie_store_dir`, обычный JSON,
   гитигнорится; в логах/выводе не печатается полное содержимое (masked/trimmed), как и
   прокси-креды.

## Что осознанно не входит в этот слайс

- Headless/Xvfb-прогрев на сервере без экрана — открытый вопрос (SPEC §9.2).
- Редактор параметров подключения, общий редактор скриптов, вкладка логов — отдельные слайсы.
- Хранилище секретов/шифрование кук — вне охвата локального режима (ADR-0009).
- Изменения в коллекторах WB/Ozon, парсере, прокси-слое, storage/IO seam, сторе городов —
  этот слайс только добавляет прогрев-оркестрацию, скрипт и вкладку поверх них.

## Последствия

- `docs/SPEC-panel.md` §4 помечается доставленной (локально); серверный прогрев остаётся
  открытым вопросом.
- `docs/ROADMAP.md`: Фаза 8.3 — готово (локально); следующий слайс — параметры подключения
  (8.4, бэкенд уже готов из `prompt-13`) либо редактор общего профиля.
- Любой будущий код, которому нужна WB-кука по региону, должен знать, что WB хранится под
  одним ключом `_session`, а не по `region_code` — обратное было бы неверной моделью данных
  (WB не зависит от куки по региону).
