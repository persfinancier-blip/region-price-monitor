# ADR-0004 — Планирование прогонов и рантайм

- **Статус:** принято (рантайм-хостинг — отложен)
- **Дата:** 2026-07-22
- **Контекст задачи:** [TZ.md](../TZ.md)

## Контекст

Замеры должны идти по расписанию 1–N раз в день, без ручного вмешательства. Масштаб (число пар товар × регион и частота) пока не зафиксирован, конкретная площадка прода — тоже. Нужна схема, которая работает на малом объёме на одном хосте и масштабируется без переписывания.

## Решение

Разделить **планирование**, **очередь задач** и **исполнение** — три роли за интерфейсами:

- **Scheduler** — по расписанию (cron-выражение) создаёт «прогон» (`run`) и ставит в очередь задачи-замеры для всех активных пар (товар × регион). На MVP — **APScheduler** внутри процесса приложения.
- **Очередь задач** — абстракция `TaskQueue`. На MVP — **очередь-таблица в PostgreSQL** (`SELECT ... FOR UPDATE SKIP LOCKED`), без отдельного брокера. Интерфейс позволяет позже переключиться на Redis + Arq/Celery без переписывания коллекторов.
- **Worker pool** — исполнители замеров: ограниченная конкурентность браузерных контекстов (asyncio semaphore) на MVP; горизонтально — несколько контейнеров-воркеров, разбирающих одну очередь.

**Рантайм:** всё пакуется в Docker; `docker-compose` для локали и MVP (app + postgres). Конкретный прод-хостинг (VPS / k8s / облако) — **отложенное решение**; портируемость контейнеров гарантирует, что выбор не потребует переписывания.

## Обоснование

- Очередь-в-Postgres убирает лишнюю инфраструктуру (Redis/брокер) на старте, но `SKIP LOCKED` уже даёт безопасную конкуренцию нескольких воркеров.
- Разделение ролей за интерфейсами = масштабирование конфигурацией, а не рефакторингом: тот же код едет с 1 воркера на N.
- APScheduler достаточно для «1–N раз в день»; при усложнении расписаний заменяется точечно.

## Альтернативы

- **Системный cron + разовый скрипт** — просто, но нет управляемости прогонами, ретраев и наблюдаемости; отклонено.
- **Celery/RQ + Redis с самого начала** — избыточно для неизвестного (вероятно малого) объёма MVP; заложено как путь роста через `TaskQueue`.
- **Kubernetes на старте** — преждевременно при нерешённом масштабе; возможно позже, контейнеры к этому готовы.

## Последствия

- Нужны сущности `runs` и очередь задач в схеме БД (см. [ARCHITECTURE.md](../ARCHITECTURE.md)).
- Порог конкурентности браузеров подбирается под ресурсы хоста; выносится в конфиг.
- Решение о хостинге и о переходе на внешний брокер принимается на фазе оркестрации/деплоя, когда прояснится масштаб.
