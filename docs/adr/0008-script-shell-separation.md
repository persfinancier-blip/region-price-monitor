# ADR-0008 — Разделение «скрипты vs оболочка»

- **Статус:** принято, не реализовано (reshapes Фазу 8)
- **Дата:** 2026-07-23
- **Контекст задачи:** [SPEC-panel.md](../SPEC-panel.md), [ADR-0006](0006-panel-and-delivery.md),
  [ROADMAP.md](../ROADMAP.md) → Фаза 8

## Контекст

Сейчас исполняемая логика (коллекторы, прогрев кук, здоровье прокси, оркестрация) живёт
внутри пакета `app` и вызывается через единый CLI (`app/cli.py`). Владелец решил (2026-07-23)
разложить эту логику на **самостоятельные, но взаимосвязанные Python-скрипты**, каждый из
которых работает и как отдельный исполняемый файл, и под управлением тонкой панели. Панель
(«Вектор·OS», SPEC-panel §6 «Редактор скриптов») получит **пайплайн в духе GitHub Actions**:
последовательности шагов с зависимостями поверх этих скриптов — оператор собирает прогон
из шагов вместо жёстко зашитого порядка.

## Решения

1. **Оболочка (панель/CLI) — только I/O и управление, без бизнес-логики.** Она читает
   конфиг, показывает статус, запускает скрипты и собирает вывод; вся логика замера/прогрева/
   здоровья остаётся в скриптах.

2. **Плановый набор скриптов:**
   - **control-panel** — города и их настройки (прокси, куки, расписания);
   - **parameters** — адреса подключения: что куда писать, что читать в локальную БД;
   - **health** — опрашивает куки/прокси; при провале триггерит прогрев кук или обновление
     прокси;
   - **wb** (и аналоги по площадкам, напр. **ozon**) — берёт параметры из control-panel/
     parameters, парсит цену;
   - **orchestrator** (рабочее имя — `runner`/`conductor`/`pipeline`, финализируется при
     реализации) — запускает последовательности скриптов, владеет порядком и стартом.

3. **Каждый скрипт работает headless самостоятельно и под оболочкой.** Никакой скрипт не
   должен требовать панель для запуска; панель — удобство, не зависимость.

4. **Редактор скриптов панели (SPEC-panel §6) получит пайплайн-конструктор** — шаги и
   зависимости между скриптами, аналогично step-графу GitHub Actions.

## Последствия

- Реструктурирует Фазу 8 (панель): «Редактор скриптов» перестаёт быть просто просмотром/
  запуском одного файла — становится конструктором пайплайна над независимыми скриптами.
- Требует извлечения текущей логики `app/*` в отдельные исполняемые модули до реализации
  панели — объём работы уточняется отдельным промптом перед Фазой 8.
- **Не реализуется в этом слайсе** (prompt-ops-01) — только зафиксировано решение.

## Открытые вопросы (follow-up)

- Финальное имя оркестратора (`runner`/`conductor`/`pipeline`).
- Формат описания пайплайна (YAML/JSON шаги + зависимости) — до реализации в Фазе 8.
- Как текущие CLI-команды (`app/cli.py`) мигрируют в новые скрипты без разрыва обратной
  совместимости для `docker-compose.prod.yml`/`Makefile` из этого слайса.
