# ADR-0007 — Наблюдаемость: метрики, структурированные логи, алерт по доле успеха

- **Статус:** принято
- **Дата:** 2026-07-23
- **Контекст задачи:** [TZ.md](../TZ.md) §Наблюдаемость / §Нефункциональные требования,
  [ARCHITECTURE.md](../ARCHITECTURE.md) §Observability, [ROADMAP.md](../ROADMAP.md) → Фаза 6

## Контекст

Фаза 5 (оркестрация) даёт устойчивый прогон, но он «немой»: только `print` в консоль,
никакой структуры, никакого сигнала о деградации доли успеха. Владелец разбил Фазу 6 на
два слайса: этот (`prompt-07`) — только наблюдаемость поверх уже работающего пайплайна;
здоровье прокси/cooldown и тонкая настройка антибота — отдельный `prompt-08`.

## Решения

1. **Метрики — производные от `runs`/`attempts`, без новой схемы.** Никакого
   отдельного хранилища метрик и exporter-демона: `app/obs/metrics.py::compute_run_metrics`
   агрегирует существующие таблицы (`attempts.queue_id → measure_queue.id`, фильтр по
   `run_id`) в `RunMetrics` (доля успеха/банов/ошибок, средняя длительность,
   `attempts_per_success` — MVP-прокси для стоимости успешного замера). Причина: не плодить
   миграции и синхронизацию состояния ради метрик, которые и так выводимы из уже
   записанных попыток.

2. **Нет живого `/metrics`-эндпоинта в этой фазе.** CLI-команда `metrics` печатает
   человекочитаемую сводку и Prometheus text-exposition формат (готовый к скрейпу текст,
   без запущенного HTTP-сервера). Постоянный эндпоинт — задача панели/деплоя (Фаза 7–8),
   когда появится процесс, которому есть смысл его обслуживать.

3. **`Alerter` — protocol-сиим, зеркалящий `ProxyProvider` (ADR-0003).** Дефолт —
   `LogAlerter` (структурный WARN в лог, работает без конфигурации). Опционально —
   `WebhookAlerter` (POST JSON на `alert_webhook_url` через уже имеющийся `requests`).
   Ни один вендор (Telegram/Slack/PagerDuty) не зашит — это забота интеграции на стороне
   вебхука, не кода. Чистая `should_alert(metrics, threshold, min_measures)` решает,
   когда стрелять; `run_once` вызывает её после финализации рана и никогда не откатывает
   ран из-за сбоя алерта.

4. **Здоровье прокси, cooldown/ротация и тонкая настройка антибота — за скобками, в
   `prompt-08`.** `StaticProxyProvider.report` остаётся no-op. Единственное требование
   к этому слайсу — чистое логирование `proxy_ref` на попытку, чтобы `prompt-08` мог
   вывести health-скор из той же таблицы `attempts` без переделки логирования.

## Обоснование

- Zero-миграционный путь — метрики и алерты можно доставить в один PR без риска для схемы
  и без блокировки на решении по будущему time-series хранилищу.
- `Alerter`-сиим повторяет уже проверенный паттерн `ProxyProvider` — предсказуемо для
  дальнейших читателей кода, дёшево добавить второй канал (просто ещё один класс).
- Stdlib-only JSON-логи и Prometheus text — ноль новых зависимостей (`prometheus_client`,
  `structlog` намеренно не добавлены).

## Последствия

- `app/obs/` — новый пакет: `logging.py` (JSON `Formatter`), `metrics.py` (`RunMetrics` +
  агрегация + Prometheus text), `alerts.py` (`Alert`, `Alerter`, `LogAlerter`,
  `WebhookAlerter`, `should_alert`, `make_alerter`).
- `run_once` эмитит `run.started`/`run.finished`; `measure_pair` эмитит одно
  `measurement`-событие на попытку — CLI и воркер-пул получают его бесплатно, так как оба
  идут через общий `measure_pair`.
- Секреты и деньги никогда не логируются: только маскированный `lease.ref`,
  `Decimal`-поля цены не попадают ни в логи, ни в payload алерта.
- `prompt-08` (здоровье прокси, cooldown, антибот-тюнинг) — следующий слайс Фазы 6,
  использует ту же таблицу `attempts` и то же поле `proxy_ref`.
