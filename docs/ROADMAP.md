# План разработки — region-price-monitor (Prizma)

План — это последовательность **вертикальных слайсов**. Каждый слайс = один промпт воркеру (`prompts/prompt-NN-*.md`) = один PR (см. [COWORK.md](../COWORK.md), [github-automation](../.claude/rules/github-automation.md)). Слайс доводится до зелёного DoD и мержа, только потом следующий.

Легенда: 🎯 цель слайса · ✅ Definition of Done (сверх общих правил: DEVLOG обновлён, гейт зелёный, смёржено в `main`).

> **Обновление 2026-07-22 (по итогам расширенного спайка и решений владельца):**
> - **Метод сбора изменён** ([ADR-0005](adr/0005-scraping-method-update.md), заменяет [ADR-0002](adr/0002-scraping-strategy.md)): WB — `requests` через прокси; Ozon — `curl_cffi` (TLS под Chrome) + **прогретые куки**; браузер Playwright — только для разового **прогрева кук**. Соответственно, коллекторы в фазах 2 и 4 строятся не на браузере, и появляется отдельный компонент **«прогрев кук»**.
> - **Панель управления и поставка добавлены в объём** ([ADR-0006](adr/0006-panel-and-delivery.md), [SPEC-panel.md](SPEC-panel.md)): после ядра сбора — **Фаза 8 «Панель» (FastAPI, стиль Вектор·OS)** и **zip-автоустановщик** под Windows/Linux (расширение Фазы 7 «Упаковка»).

## Фаза −1 — Спайк осуществимости (гейт GO/NO-GO) · `prompt-00-spike`

🎯 **Прежде чем брать разработку в работу** — проверить рискованные механизмы, а не обвязку: (1) заход с преднастроенным гео через региональный прокси работает; (2) цена одного и того же SKU меняется по региону; (3) цена конкретного SKU читается на WB и Ozon; (4) антибот проходится (капча детектится и обходится сменой прокси). Throwaway: без БД, расписания и чистой архитектуры — один скрипт и отчёт.
📋 **Нужно от владельца:** триал прокси (≥2 региона РФ), 2–3 образцовых SKU на каждый маркетплейс, список нужных ценовых полей.
✅ `docs/spike-report.md`: таблица SKU × регион × цена, PASS/FAIL по механизмам 1–4, доля блоков, вердикт **GO / NO-GO**. Продовый код из спайка не мержится. На GO — стартуем с `prompt-01`.

## Фаза 0 — Скелет и тулинг · `prompt-01-skeleton`

🎯 Каркас репозитория: структура пакета, `pyproject.toml`, ruff/mypy/pytest, `scripts/dod.sh` с реальными командами, `docker-compose` (app + postgres), конфиг (pydantic-settings), подключение к БД, `.env.example`, инициализация Alembic.
✅ `scripts/dod.sh` реально гоняет линт+типы+тесты и зелёный; `docker compose up` поднимает app и postgres; пустой `alembic upgrade head` проходит.

## Фаза 1 — Модель данных и миграции · `prompt-02-schema`

🎯 Таблицы `products`, `regions`, `runs`, `measure_queue`, `price_snapshots`, `attempts` (см. [ARCHITECTURE.md](ARCHITECTURE.md)); SQLAlchemy-модели, репозитории, первая Alembic-миграция; CLI-импорт справочников товаров и регионов.
✅ Миграция применяется/откатывается; CLI грузит демо-набор товаров и регионов; тесты репозиториев зелёные.

## Фаза 2 — Коллектор WB, один регион, без прокси · `prompt-03-wb-collector`

🎯 Интерфейс `MarketplaceCollector` + `WbCollector` на `requests` (ADR-0005, эндпоинт `card.wb.ru`): по товару WB снять цену/наличие и сохранить `price_snapshot`. Сквозной путь «товар → запрос → БД» на одном (домашнем) регионе, ещё без прокси.
✅ Ручной прогон снимает и пишет реальную цену WB; парсер покрыт тестом на зафиксированном сэмпле ответа.

## Фаза 3 — Регионализация + ProxyProvider · `prompt-04-regions-proxy`

🎯 Интерфейс `ProxyProvider` + `StaticProxyProvider` (тест/локаль) и обвязка под коммерческого; гео-параметры региона в браузер; WB-замеры по нескольким регионам через прокси; учёт исхода (`report`).
✅ Один товар WB даёт различающиеся региональные цены; смена прокси/гео отражается в замере; `attempts` фиксируют исход.

## Фаза 4 — Коллектор Ozon · `prompt-05-ozon-collector`

🎯 `OzonCollector` на `curl_cffi` + прогретые куки (ADR-0005); отдельный компонент **прогрева кук** (Playwright, разовое решение капчи, сохранение `storage_state`, хранилище кук по «город × площадка»); регион — через региональную куку; те же поля цены/наличия; интеграция в общий поток.
✅ Ozon-замеры проходят по нескольким регионам через куки; детект протухания кук (403) ведёт к повторному прогреву; парсер покрыт тестом на сэмпле.

## Фаза 5 — Оркестрация и расписание · `prompt-06-orchestration`

🎯 `Scheduler` (APScheduler) + `TaskQueue` (очередь-в-Postgres, `SKIP LOCKED`) + worker pool с ограниченной конкурентностью; жизненный цикл `run`; ретраи с backoff, обработка банов; идемпотентность прогона.
✅ Одна команда запускает полный прогон по всем активным парам; несколько воркеров безопасно делят очередь; `runs.stats` агрегирует итоги.

## Фаза 6 — Устойчивость и наблюдаемость

Разбита владельцем на два слайса: 6.1 — наблюдаемость (метрики/логи/алерт), 6.2 — здоровье
прокси и антибот-тюнинг. Оба опираются на `attempts`/`app/obs` без новой схемы (ADR-0007).

### Фаза 6.1 — Наблюдаемость · `prompt-07-observability` — ✅ готово

🎯 Метрики (успех/бан/ошибки, стоимость на успешный замер), структурные логи, алерт при просадке доли успеха.
✅ Метрики прогона доступны; искусственный бан-сценарий корректно ретраится и отражается в метриках; порог алерта срабатывает.

### Фаза 6.2 — Здоровье прокси и антибот · `prompt-08-proxy-health`

🎯 Политика здоровья/cooldown прокси, выведенная из `attempts` (без новой схемы) — забаненный
прокси остывает, а не долбится дальше; тонкая настройка антибота: паттерн-темп запросов
(мин. интервал + джиттер на маркетплейс) и консистентный по региону fingerprint (WB-заголовки,
Ozon `impersonate`).
✅ `HealthAwareProxyProvider` останавливает измерение остывающего прокси без фейкового `attempts`;
паттерн-темп общий на пул воркеров; fingerprint детерминирован по региону; DoD-гейт зелёный.

## Фаза 7 — Упаковка и деплой · `prompt-ops-01-deploy`

🎯 Финализация Dockerfile (Playwright-база), прод-`compose`, ops-документация, выбор хостинга (снятие открытого вопроса), расписание в проде.
✅ Образ собирается и запускает прогон по расписанию в целевой среде; ops-доки описывают запуск, секреты, обновление.

## Порядок и зависимости

Фаза −1 (спайк) — **гейт**: разработка (фазы 0→7) начинается только при вердикте GO. Фазы 0→7 строго последовательны (каждая опирается на предыдущую), кроме упаковки: черновой Dockerfile/compose появляется уже в фазе 0, финализируется в фазе 7; внутри фазы 6 слайс 6.1 (наблюдаемость) предшествует 6.2 (здоровье прокси) — оба читают одну и ту же таблицу `attempts`. Триал прокси нужен **уже для спайка**; финальный выбор провайдера — перед фазой 3. Масштаб уточняется **перед фазой 5** (влияет на выбор «очередь-в-БД vs брокер»). Хостинг — **перед фазой 7**.
