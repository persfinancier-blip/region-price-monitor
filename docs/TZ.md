# ТЗ — region-price-monitor

> **Кодовое имя:** **Prizma** (Призма).
> Одним словом суть сервиса — **ценоразведка**: он «раскладывает» цену товара по регионам РФ, как призма — свет по спектру.

## Цель

Отслеживать цены на заданный список товаров на **Wildberries** и **Ozon** **в разрезе регионов РФ**. Для каждого товара и каждого целевого региона регулярно (раз или несколько раз в день) снимать актуальную цену так, как её видит покупатель именно из этого региона, и сохранять историю замеров.

Ключевая сложность: маркетплейсы показывают разные цены в зависимости от геолокации покупателя и активно защищаются от автоматического сбора. Значит, сервис должен:

- ходить на маркетплейсы **через региональные прокси**, чтобы получать «региональную» цену;
- **устойчиво проходить антибот-защиту** маркетплейсов;
- работать **по расписанию** (1–N раз в день) без ручного вмешательства.

Список товаров и регионов сервис берёт из PostgreSQL; результаты замеров пишет туда же.

## Стек (зафиксировано в ADR)

- **Язык:** Python 3.12 (строго Python — требование владельца).
- **Сбор данных (обновлено):** WB — `requests` через региональный прокси; Ozon — `curl_cffi` (TLS под Chrome) + прогретые куки; браузер **Playwright** — только для разового прогрева кук Ozon. См. [ADR-0005](adr/0005-scraping-method-update.md) (заменяет [ADR-0002](adr/0002-scraping-strategy.md)).
- **Панель и поставка:** локальная веб-панель на **FastAPI** (стиль «Вектор·OS») + zip-автоустановщик под Windows и Linux. См. [ADR-0006](adr/0006-panel-and-delivery.md), [SPEC-panel.md](SPEC-panel.md).
- **Хранилище:** **PostgreSQL 16**, доступ через SQLAlchemy 2.x, миграции — Alembic.
- **Прокси:** коммерческий провайдер резидентных/мобильных прокси с гео-таргетингом по РФ, спрятанный за интерфейсом `ProxyProvider`. См. [ADR-0003](adr/0003-proxy-provider.md).
- **Планировщик/рантайм:** контейнеры (Docker), портируемые между хостами; планировщик внутри приложения. Конкретный хостинг — позже. См. [ADR-0004](adr/0004-scheduling-runtime.md).
- **Тулинг:** ruff (линт+формат), mypy (типы), pytest (тесты), pydantic-settings (конфиг).

Полное обоснование выбора — [ADR-0001](adr/0001-stack.md). Архитектура и модель данных — [ARCHITECTURE.md](ARCHITECTURE.md). План работ — [ROADMAP.md](ROADMAP.md).

## Функциональные требования

1. **Каталог товаров.** Хранить список отслеживаемых товаров: маркетплейс (WB/Ozon), артикул/SKU, URL, человекочитаемое имя, признак активности.
2. **Справочник регионов.** Хранить список целевых регионов с гео-параметрами, нужными каждому маркетплейсу (для WB — параметр доставки/`dest`; для Ozon — город/координаты/адрес доставки).
3. **Замер цены.** Для пары (товар × регион) получить: текущую цену, цену без скидки, цену «с картой»/кошельком (если применимо), доступность/наличие, а также сырой ответ (raw) для отладки.
4. **Региональность.** Замер должен отражать цену **именно для указанного региона** — через прокси нужного региона и/или гео-параметры маркетплейса.
5. **История.** Каждый замер — отдельная запись со временем; старые значения не перезаписываются (нужна динамика цен).
6. **Расписание.** Полный прогон по всем активным парам запускается по расписанию 1–N раз в день; частота настраивается.
7. **Устойчивость к антиботу.** Ротация прокси, реалистичные отпечатки браузера, человекоподобный темп, ретраи с backoff, детект капчи/блокировки → повтор через другой прокси.
8. **Наблюдаемость.** По каждому прогону — статистика: успешно/ошибки/заблокировано, доля успеха, доля банов. Логи структурированные.
9. **Загрузка данных.** CLI/скрипт для наполнения справочников товаров и регионов (импорт).

## Нефункциональные требования

- **Надёжность сбора:** целевая доля успешных замеров за прогон — задаётся как KPI (ориентир ≥ 90% на MVP); при просадке — алерт.
- **Изоляция сбоев:** падение одного замера не роняет прогон; каждый замер ретраится независимо.
- **Идемпотентность прогона:** повторный запуск не портит данные, дубли отсекаются.
- **Портируемость:** всё в контейнерах, без привязки к конкретному облаку.
- **Масштабируемость:** архитектура рассчитана на рост числа пар (товар × регион) без переписывания — через пул воркеров и очередь. Стартуем с малого объёма.
- **Секреты не в коде:** креды прокси и БД — только через окружение/секрет-хранилище; `.env` в `.gitignore`.
- **Вежливость сбора:** ограничение частоты запросов к маркетплейсам, только публично доступные данные о ценах.

## Не делаем (границы MVP)

- **UI-панель управления — теперь В ОБЪЁМЕ** (решение владельца 2026-07-22): панель управляет городами, профилями площадок, куками, маппингом данных, логами. См. [SPEC-panel.md](SPEC-panel.md), [ADR-0006](adr/0006-panel-and-delivery.md). (Визуализация динамики цен — по-прежнему за MVP.)
- Не работаем с личными кабинетами продавца и приватными API маркетплейсов — только публичные цены.
- Не отслеживаем остатки на складах, отзывы, рейтинги, позиции в поиске — только цена и доступность (расширяемо позже).
- Не поддерживаем другие маркетплейсы, кроме WB и Ozon (интерфейс парсера оставляем расширяемым).
- Не оформляем заказы, не кладём в корзину сверх необходимого для получения цены.

## Открытые вопросы

- **Масштаб.** Точный объём (число SKU × регионов × частота) пока не задан → закладываем масштабируемую абстракцию, MVP гоняем на малом объёме. Уточнить перед фазой оркестрации.
- **Хостинг прода.** Конкретная площадка (VPS / k8s / облако) — решаем позже; до этого держим всё портируемым в контейнерах.
- **Провайдер прокси.** Конкретный вендор резидентных прокси не выбран → кодим против интерфейса `ProxyProvider`, вставим реализацию перед регионализацией.
- **Список регионов.** Финальный перечень целевых регионов и их гео-параметры для каждого маркетплейса — уточнить у владельца.
- **Ценовые поля.** Полный список нужных полей цены (базовая / со скидкой / с картой WB / Ozon Premium и т.п.) — уточнить, влияет на схему `price_snapshots`.
- **Региональная кука Ozon.** Определить, какая именно кука несёт регион (эксперимент), чтобы задавать регион программно. См. [ADR-0005](adr/0005-scraping-method-update.md).
- **Срок жизни кук Ozon.** Измерить (частота прогрева зависит от него) — можно параллельно разработке.
- **Прогрев кук на Linux-сервере.** Локальный прогрев + перенос vs headful через Xvfb. См. [ADR-0006](adr/0006-panel-and-delivery.md).
