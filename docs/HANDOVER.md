# region-price-monitor — передача проекта

Документ для владельца проекта. Объясняет, что система делает, в каком она сейчас
состоянии, и описывает интерфейс: схему базы данных, команды запуска, компоненты и
внешние зависимости. Технические детали даны в объёме, достаточном чтобы принять
работу и передать её разработчику на сборку боевого пайплайна.

## 1. Что делает система

Отслеживает цены заданного списка товаров на **Wildberries** и **Ozon** в **разных
регионах РФ** и складывает замеры в PostgreSQL. Запускается по расписанию (раз или
несколько раз в день). Главная сложность — защита маркетплейсов от автоматического
сбора; ниже описано, как она обойдена.

Вход и выход — обе в PostgreSQL: на входе таблица со списком товаров и регионов, на
выходе таблица с историей цен. Всё остальное (прокси, куки, обход антибота) — внутренняя
кухня, наружу торчит только база.

## 3. Как обойдена защита (суть)

**Wildberries** — простой случай: цена берётся обычным HTTP-запросом (`requests`) через
региональный прокси. Браузер не нужен.

**Ozon** — сложный случай, защита «ABT» (капча-пазл + блокировка по репутации IP +
проверка отпечатка клиента). Рабочая связка из трёх частей:

1. **Прогрев кук.** Один раз в настоящем браузере проходится антибот/капча, и
   сохраняются полученные куки (технические, не логин от аккаунта).
2. **Быстрые запросы через `curl_cffi`.** Библиотека маскирует TLS-отпечаток под
   Chrome. Без неё обычный `requests` получает `403`, даже с валидными куками — Ozon
   опознаёт «непохожий на браузер» запрос. С ней — отдаёт цену как обычному Chrome.
3. **Совпадение IP.** Запрос идёт с того же IP, где грелись куки.

**Регион** зашит в куках: меняя региональную куку, можно получать цены десятков
регионов **с одного IP** — парк региональных прокси для чтения Ozon не обязателен.
Проверено: смена города доставки меняет цену при `HTTP 200`.

Экономический смысл: то же, что делают платные скрейпинг-сервисы, реализовано своими
силами. Плата за это — периодический «прогрев» кук (см. открытые вопросы).

## 4. Интерфейс

### 4.1. Схема PostgreSQL (целевая)

Вход — что мониторить:

```
products                        -- список товаров
  id            bigserial PK
  marketplace   text            -- 'wb' | 'ozon'
  sku           text            -- артикул/ID товара на площадке
  url           text            -- прямая ссылка на карточку
  name          text
  is_active     boolean         -- мониторить или нет
  created_at    timestamptz

regions                         -- регионы РФ для замеров
  code          text PK         -- напр. 'msk', 'spb', 'vvo'
  name          text            -- 'Москва', 'Санкт-Петербург', ...
  ozon_cookie   text            -- региональная кука Ozon (какая именно — см. п.6)
  proxy_ref     text            -- ссылка на прокси для этого региона (для WB)
```

Выход — история цен:

```
price_measurements              -- один замер = одна строка
  id            bigserial PK
  product_id    bigint FK -> products.id
  region_code   text FK -> regions.code
  measured_at   timestamptz     -- момент замера
  price         numeric          -- текущая цена
  price_no_card numeric          -- цена без карты (перечёркнутая/обычная)
  price_card    numeric          -- цена с картой (Ozon-карта / WB-кошелёк)
  currency      text             -- 'RUB'
  status        text             -- 'ok' | 'blocked' | 'not_found'
```

Замеры **не перезаписываются** — копится история, по ней строится динамика цен.

### 4.2. Команды и конфигурация

Секреты — через переменные окружения и `.env` (в репозиторий не коммитятся):

- `DATABASE_URL` — строка подключения к PostgreSQL.
- `PROXY` — прокси в формате `host:port:login:password` (ASocks, схема `https`).
- `OZON_COOKIE` — прогретая cookie-строка Ozon (на регион).

Проверочные скрипты этапа спайка (в папке `spike/`):

```
# WB — цена через прокси
python spike/check_wb.py

# Ozon — разовый прогрев кук в браузере (решить капчу руками)
set MANUAL=1 & set FIND_MOBILE=1
python spike/check_ozon.py

# Ozon — быстрый замер цены по прогретым кукам (боевой способ чтения)
python spike/check_ozon_cookies.py
```

Боевой запуск (целевой, после сборки пайплайна) — одна команда чтения по расписанию
(cron / GitHub Actions / локально), которая читает `products` + `regions`, снимает цены
и пишет в `price_measurements`.

### 4.3. Компоненты (границы модулей)

- **WB-фетчер** — `requests` + региональный прокси, парсит цену WB.
- **Ozon-фетчер** — `curl_cffi` (Chrome TLS) + прогретые куки, парсит цену Ozon; регион
  задаётся региональной кукой.
- **Прогрев кук (Ozon)** — Playwright + Chromium, ручное решение капчи, сохраняет куки/
  сессию по региону. Запускается редко (по мере протухания кук), не в каждом замере.
- **Слой БД** — читает `products`/`regions`, пишет `price_measurements`.
- **Планировщик** — запускает замеры по расписанию.
- **Менеджер регионов** — сопоставляет регион → кука Ozon / прокси WB.

### 4.4. Внешние зависимости

- **PostgreSQL** — хранилище входа и выхода.
- **`curl_cffi`** — критично для Ozon: маскировка TLS-отпечатка под Chrome. Без неё Ozon
  отдаёт `403`.
- **Прокси ASocks** — резидентные/мобильные IP РФ. Нужны для WB (региональность) и для
  прогрева кук Ozon. Особенность: это HTTPS-прокси (TLS до прокси, самоподписанный серт —
  требуется отключение проверки сертификата; схема `https`).
- **Playwright + Chromium** — только для разового прогрева кук Ozon (решение капчи).
  В рутинных замерах не используется.

## 6. Открытые вопросы

1. **Срок жизни кук Ozon** — измерить (прогонять чтение через час/сутки, зафиксировать,
   когда прилетит `403`). От этого зависит частота прогрева.
2. **Какая именно кука несёт регион** — вычислить конкретное имя/значение, чтобы задавать
   регион программно (сейчас доказан сам факт, что регион — в куках).
3. **Автоматизация прогрева** — сейчас решение капчи ручное; продумать, как минимизировать
   ручные заходы (например, один прогрев на регион в сутки).
4. **Реакция на смену защиты Ozon** — при изменении ABT метод может потребовать правок;
   заложить мониторинг доли `blocked` в замерах.
